Campaign assets

캠페인 자산은
전환을 향해 정렬됩니다.

캠페인별 사이트는 같은 템플릿을 복제하는 대신, 고객 여정에서 맡은 역할에 따라 정보 구조와 CTA가 달라집니다. T1은 리드를 직접 받고, T2는 관심을 설명하고 대표 T1으로 전달합니다.

T1대표 오퍼, 신뢰 정보, 최종 문의. 캠페인의 지정 리드 이메일로 직접 연결.
T2비교, 해설, 문제 정의와 사전 설득. 문의 CTA는 캠페인 대표 T1으로 라우팅.

Shared assets

공용 자산은
별도의 리듬으로 축적됩니다.

T3 공용 풀은 특정 캠페인의 생성 큐에 종속되지 않습니다. 다양한 니치와 페르소나로 콘텐츠를 축적하고, 링크주스 스케줄러가 전체 캠페인의 상위 자산을 향해 분산 연결합니다.

T3발견과 분산을 위한 공용 니치 자산. 캠페인 자산 생성 파이프라인과 분리 운영.
SNS계정별 활동 한도와 쿨다운을 지키며 T3 중심으로 순환하고 일부 T2를 보조.

03 / Architecture

오케스트레이션은 판단하고,
실행기는 기록을 남깁니다.

로컬 LLM은 캠페인 문맥과 자원 상태를 읽어 작업을 조합합니다. 실제 생성·배포·검증은 정해진 실행기와 큐를 통하며, 사용자는 현재 어떤 단계가 진행 중인지 대시보드에서 확인합니다.

ROLE BOUNDARY

사이트 생성 잡과 링크주스·웜업 잡은 목적과 우선순위에 따라 분리 실행됩니다.

CAMPAIGN ISOLATION

전략, 콘텐츠, 리드 목적지와 할당 자원은 캠페인 문맥을 벗어나지 않습니다.

PERMANENT HOST

한 번 정한 사이트·호스트·공개 URL 관계는 재배포에서도 유지합니다.

FAILURE RECOVERY

실패 작업은 상태를 보존하고, 원인을 수정한 뒤 같은 목적지를 향해 재시도합니다.

04 / Decision guide

운영 방식이 맞아야
시스템이 효율적입니다.

PBN-BUILDER는 버튼 하나로 결과를 보장하는 도구가 아니라, 여러 캠페인의 자산과 실행을 지속적으로 관리하려는 운영자에게 맞춰져 있습니다.

적합한 운영

  • 여러 사업 또는 브랜드 캠페인을 동시에 관리한다.
  • 사이트 제작 이후 콘텐츠와 링크 운영까지 연결해야 한다.
  • 로컬 LLM을 사용하며 외부 API 의존도를 낮추고 싶다.
  • 캠페인 자산과 공용 자산을 분리해 축적하려 한다.
  • 실패와 재시도 상태까지 기록되는 운영 도구가 필요하다.

맞지 않는 기대

  • 검색 순위나 매출의 즉시 달성을 보장받고 싶다.
  • 검토 없이 모든 계정과 채널을 자동 실행하려 한다.
  • 하나의 템플릿을 대량 복제하는 것만 필요하다.
  • 호스트, 도메인, 세션 자원을 임의로 교체하려 한다.
  • 운영 상태 확인 없이 결과 파일만 받으려 한다.
